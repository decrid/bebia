# Security policy

## Supported versions

Security reports are evaluated against the latest source on `main` and the
latest release published by the project, if one exists. Older snapshots are not
guaranteed to receive fixes. The repository currently does not declare a
long-term support schedule.

## Reporting a vulnerability

Do not publish sensitive vulnerability details in a public issue, discussion,
pull request, commit, screenshot, or CI log.

Use GitHub Private Vulnerability Reporting from this repository's
**Security → Advisories → Report a vulnerability** flow. This creates a private
draft advisory visible to repository maintainers. Include:

- affected version or commit,
- reproducible steps with secrets and personal data removed,
- expected and observed impact,
- relevant platform and device details,
- a minimal proof of concept where safe.

If the private reporting button is unavailable, do not fall back to a public
issue. The maintainers must first enable a private reporting channel. No
security email address is documented in this repository, so this policy does
not invent one.

Please allow maintainers time to validate and coordinate a fix before public
disclosure. Never include a real child profile, family invitation, signing
material, Firebase credential, token, or recorded audio in a report.
